Документ предназначен для примера использования ряда подсистем.

В подсистеме Бизнес-процессы и задачи на основании документа можно ввести бизнес-процесс Задание, он также может выступать предметом задач.

В подсистеме Даты запрета изменения документ показывает пример запрета изменения документа по разделу дат запрета Демо: Торговля с объектом Демо: Группа доступа партнеров, который получается, как реквизит Демо: Группа доступа покупателей объекта Демо: Партнер. При этом реализован пример для различных сложных случаев получения даты и объекта при проверке документа, для чего используются по-очереди:

  • реквизит Дата и реквизит Партнер;
  • реквизит Дата и реквизит Партнер табличной части Партнеры и контактные лица (проверка для всех партнеров табличной части);
  • реквизит Дата и реквизит Партнер табличной части Партнеры и контактные лица (проверка для всех партнеров табличной части);
  • реквизит Счета на оплату.Дата и реквизит Партнер табличной части Партнеры и контактные лица (проверка для всех партнеров табличной части);
  • реквизит Счета на оплату.Дата и реквизит Партнер табличной части Партнеры и контактные лица (проверка для всех партнеров табличной части);
  • реквизит Счета на оплату.Дата и реквизит Счета на оплату.Счет.Партнер.

 

В подсистеме Взаимодействия документ выступает предметом взаимодействий.

В подсистеме Запрет редактирования реквизитов объектов документ показывает пример того, как реквизиты Организация, Партнер, Контрагент, Договор и табличная часть Счета на оплату могут быть заблокированы для изменения.

В подсистеме Контактная информация документ иллюстрирует возможность добавления произвольных полей контактной информации на форму (см. закладку Доставка) и в табличную часть Контактные лица. Поля Адрес, Телефон и Электронная почта добавляются в табличную часть Контактные лица с помощью настройки видов контактной информации. Поля Адрес и Электронная почта на закладке Доставка реализованы программно в коде модуля формы.

Для демонстрации подсистемы Обновление версии ИБ в модуле менеджера документа представлен пример отложенного обработчика обновления, который инициализирует новый реквизит СтатусЗаказа. См. процедуры ЗарегистрироватьДанныеКОбработкеДляПереходаНаНовуюВерсию и ОбработатьДанныеДляПереходаНаНовуюВерсию в модуле менеджера. Этот обработчик обновления выполняется в многопоточном режиме. В процедуре ЗарегистрироватьДанныеКОбработкеДляПереходаНаНовуюВерсию описываются параметры выборки данных. Сама выборка выполняется механизмом обновления, а не кодом в обработчике обновления. В процедуре ОбработатьДанныеДляПереходаНаНовуюВерсию выбранные механизмом обновления данные получаются вызовом ОбновлениеИнформационнойБазы.ДанныеДляОбновленияВМногопоточномОбработчике().

В подсистеме Префиксация объектов к коду элементов справочника добавляется префикс информационной базы.

В подсистеме Работа с файлами документ показывает пример объекта-владельца файлов вложений. Для хранения файлов-вложений используется общий справочник Файлы. Кроме того, в форме элемента приведен пример отображения гиперссылки присоединенных файлов.

В подсистеме Структура подчиненности документ показывает пример иерархической взаимосвязи документов (реквизит Счет табличной части Счета на оплату ссылается на документ Демо: Счет на оплату покупателю).

В подсистеме Текущие дела документ выступает в качестве поставщика текущих дел по заказам.

В подсистеме Управление доступом документ показывает пример типичного использования логики ограничения по двум видам доступа. Кроме этого, документ записывает набор значений доступа, который используется бизнес-процессом Демо: Задание с ролевой адресацией.

  • Список ограничивается по двум видам доступа:
    • Организации (проверяется значение реквизита Организация);
    • Демо: Группы партнеров (проверяется значение реквизита Партнер).
    • все значения должны быть разрешенными одновременно (в одной из групп доступа).
